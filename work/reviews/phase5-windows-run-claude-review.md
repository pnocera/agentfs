## Findings

- [low] cli/src/cmd/run_windows.rs:57-69 - For a reused `--session ID`, the run path always calls `overlay.init(cwd_str)` and overwrites `base_path` in the delta DB and in `run_dir/base_path` with the *current* working directory. If a user runs `agentfs run --session X` from cwd `A`, then later from cwd `B`, the delta layer's recorded base silently flips from `A` to `B` while the existing whiteouts/origins were written against `A`. Mirrors the macOS template, so consistent — but worth a one-line check (compare existing base_path to current cwd, bail if they diverge) or at least a doc note that re-using a session means "same cwd."
- [low] cli/src/cmd/run_windows.rs:172-188 - `find_unused_drive_letter` treats `!root.exists()` as "available." That's right for the common case but classifies a drive in a temporarily-disconnected state (network drive reconnecting, mounted CD without media) as available, which would then make `mount.exe` fail. The two unit tests cover the happy paths only; not a blocker since the resulting `mount.exe` error is actionable.
- [low] cli/src/cmd/run_windows.rs:156-170 - On mount failure the function leaves `run_dir`, `delta.db`, and the partially-written overlay metadata on disk because the early `bail!` from `mount_fs(...)` exits before any cleanup branch fires. Intentional for session reuse, but combined with the persisting Network Error 53 on this machine it means every failed `agentfs run` leaves a junk directory under `%USERPROFILE%\.agentfs\run\`. Worth a follow-up cleanup heuristic (or just a doc note that failed runs must be pruned manually).
- [low] work/phase5-cli-smoke-windows-run.log:3-13 - Live smoke is blocked by the same Phase 0-4 Windows Client for NFS `Network Error - 53` against the localhost NFS server. The plan acceptance criterion "writes into the AgentFS delta layer, not the original working tree" therefore can't be verified end-to-end on this machine; the smoke harness output `base created.txt exists: False` is consistent with "no mount, so no write happened anywhere" rather than "write went to delta." The Phase 5 spike calls this out honestly as a carry-over blocker, not a Phase 5 regression. No action required for Phase 5 itself.
- [low] cli/src/cmd/run_windows.rs:213-241 - On Windows, Ctrl+C in the parent console delivers `CTRL_C_EVENT` to the entire console process group, so the child typically receives the signal too and exits on its own. The explicit `child.kill().await` after the signal is defensive but redundant when the child handles Ctrl+C, and `kill` uses `TerminateProcess` which is forceful (no chance for the child to flush). Acceptable for v1; a more graceful sequence would race a short `child.wait()` with a kill timeout. Not a blocker.
- [info] work/phase5-cargo-check-linux-cli-default-features.log:123-165 - Linux default-features build fails on `unwind-sys v0.1.4` because `libunwind-ptrace.pc` isn't installed in this WSL environment. The dependency comes from `reverie-process` via the Linux sandbox stack and pre-dates Phase 5 — Phase 5 didn't touch the Linux sandbox path. The Linux *no-default-features* check passes (`work/phase5-cargo-check-linux-cli-no-default-features.log`), confirming Phase 5's changes don't break Linux compilation; the default-features regression is the documented environment gap and is not a Phase 5 issue.
- [info] cli/src/cmd/run_windows.rs:121-146 - `validate_windows_run_options` correctly errors for `--experimental-sandbox` and `--strace`, warns and ignores for `--allow`, `--no-default-allows`, and `--system`. Validation runs before any AgentFS setup so the user gets the right feedback fast.
- [info] cli/src/cmd/run_windows.rs:148-188 - Session layout matches the plan: `%USERPROFILE%\.agentfs\run\<session-id>\{delta.db, base_path}` with mount root from `find_unused_drive_letter()`. `Z:\` through `D:\` iteration via `(b'D'..=b'Z').rev()` is concise and the two unit tests (`chooses_unused_drive_from_high_to_low`, `returns_none_when_no_drive_is_available`) cover the boundary cases.
- [info] cli/src/cmd/run_windows.rs:194-241 - Banner + env vars + child spawn match the spike: banner explicitly says "This is not a security sandbox", lists the mount mapping and session dir, and sets `AGENTFS=1`, `AGENTFS_SANDBOX=windows-overlay-only`, `AGENTFS_SESSION=<id>`, and `PROMPT=[agentfs] $P$G`. The child uses `current_dir(&session.mount_root)`, so when the mount succeeds the child's cwd is the mounted drive root, exactly as the plan called for.
- [info] cli/src/cmd/run_windows.rs:243-280 - `prepare_windows_command` rewrites argv only when `args` is empty *and* the command basename matches `cmd[.exe]` or `powershell[.exe]`/`pwsh[.exe]`. Explicit non-shell commands and explicit shell-with-args invocations are passed through verbatim — matches the plan's "explicit non-shell commands rely on environment variables rather than rewriting user arguments." Three tests lock this in: `prepares_interactive_cmd_prompt`, `prepares_interactive_powershell_prompt`, `leaves_explicit_shell_args_unchanged`.
- [info] cli/src/cmd/run.rs:11-18 - The platform dispatch is now uniform: `run_linux.rs`, `run_darwin.rs`, `run_windows.rs` are all activated by `(target_os = X, feature = "sandbox")`, with `run_not_supported.rs` as the `not(feature = "sandbox")` fallback. That's why the Phase 4 `run_not_supported.rs:9 too_many_arguments` clippy warning is gone in `work/phase5-cargo-clippy-windows-cli-default-features.log` — Phase 5 build no longer compiles the stub on Windows default-features.

- [info] cli/src/opts.rs:133-184 - Help text and per-flag descriptions explicitly call out Windows v1 behavior: "Ignored on Windows v1 because no OS sandbox is enforced" on `--allow`, "No effect on Windows v1" on `--no-default-allows`, "ignored on Windows v1" on `--system`, and the command-level docstring states "Windows v1 uses overlay-only execution and is not a security sandbox." - [info] cli/src/cmd/run_windows.rs:87-119 - Cleanup ordering on success: `remove_proc_file(...)`, `remove_dir(procs_dir)`, `drop(mount_handle)`, then user-facing diff/session hints, then `std::process::exit(exit_code)`. `drop(mount_handle)` runs *before* `process::exit`, so the unmount and server abort happen via the Phase 2 `MountHandle::drop` Windows path. On the error branch, the early `return Err(err)` lets `mount_handle` drop at scope exit, so cleanup still runs.

All pass per `work/phase5-cargo-test-windows-cli-lib-default-features.log:22-26`.

The previous `run_not_supported.rs:9` warning is gone because Phase 5's `run_windows.rs` is the active path on Windows default-features and uses `#[allow(clippy::too_many_arguments)]`.
