## Findings

- [medium] work/phase1-nfs-compile-path.md:60-62 - The Phase 1 plan acceptance in `work/plan/implementation-plan.md:79` explicitly requires "Linux/macOS checks still pass," but the spike states "Linux/macOS checks were not run locally because this phase was executed on the Windows host." With CI workflows removed (`.github/workflows/rust.yml` deleted at base) there is also no automated coverage backstopping this. Phase 1 acceptance is therefore not literally met — a Linux `cargo check`/`cargo test` and a macOS `cargo check` (test would be ideal) must be captured before Phase 1 is declared done, because the changes touch shared SDK code (`OPEN_*` constants, FileSystem::open doc) and the cross-platform device-encoding helpers.
- [medium] cli/src/nfs.rs:27-42 vs sdk/rust/src/filesystem/hostfs_darwin.rs:167 - The new `device_major` / `device_minor` / `make_device` helpers implement glibc's `gnu_dev_*` bit layout (verified bit-for-bit), but macOS HostFS passes `stat.st_rdev` straight through into `Stats.rdev` (`sdk/rust/src/filesystem/hostfs_darwin.rs:167`) and macOS uses a different `dev_t` packing (8-bit major in bits 24-31, 24-bit minor in bits 0-23). Before Phase 1 the NFS adapter called `libc::major/minor` which on macOS matched the macOS-encoded `st_rdev`; after Phase 1 the adapter uniformly decodes with glibc semantics, so device-file `getattr`/`readdir_plus` over an NFS-served macOS HostFS now returns wrong major/minor to clients. Round-trip *inside* the NFS adapter (mknod write → getattr read) is internally consistent, so the practical impact is limited to HostFS-backed device files on macOS — a rare case, but a real semantic regression. Either normalize `Stats.rdev` at the macOS HostFS boundary, or document the new SDK contract (`Stats.rdev` is glibc-encoded) at `sdk/rust/src/filesystem/mod.rs:131` and file the macOS HostFS normalization for Phase 4.
- [low] cli/src/mount/mod.rs:117-120 - `MountHandle::drop` still calls `std::env::set_current_dir("/")` unconditionally. On Windows this resolves to the root of the current drive, which may *be* the drive the handle is dropping; the EBUSY-avoidance intent then collapses. The plan's Phase 2 task list (`work/plan/implementation-plan.md:90`) defers the Windows branch, and today no Windows caller can construct `MountHandleInner::Nfs` (there is no Windows `mount_fs`), so this is dormant. Flag it so the Phase 2 author wires the temp-dir branch *before* introducing a Windows `mount_fs`, not after.
- [low] cli/src/mount/nfs.rs:90-129 - `mount_nfs` is compiled on Windows (`#[cfg_attr(target_os = "windows", allow(dead_code))]`) and would, if called, bind the NFS TCP listener, spawn a `tokio::task`, sleep 100 ms, then bail in the `nfs_mount` Windows stub at line 178 — leaking the spawned task and the bound port. Dormant today (no Windows `mount_fs`), but Phase 2 must order the work as "preflight client detection → bind server → invoke mount.exe → on failure, cancel the server task and return," not the current Linux/macOS shape that assumes `nfs_mount` cannot fail catastrophically.
- [low] cli/src/mount/nfs.rs:81-88 + 175-183 - The Windows `unmount_nfs` and `nfs_mount` stubs both `anyhow::bail!`. That's fine for Phase 1, but the messages should be aligned with what Phase 2 will produce so call-site error formatting doesn't churn — consider phrasing the stub messages as "Windows NFS support lands in Phase 2" so a grep for "not implemented" turns up the right tracking artifact.
- [info] sdk/rust/src/filesystem/mod.rs:80-89 / sdk/rust/src/lib.rs:23 - `OPEN_READONLY=0`, `OPEN_WRITEONLY=1`, `OPEN_READWRITE=2` are added, documented with the POSIX-passthrough rationale, and re-exported. `OPEN_WRITEONLY` is bonus relative to the plan (which lists only `READONLY=0`/`READWRITE=2`); harmless and matches POSIX `O_WRONLY=1`. The shared rustdoc paragraph attaches to `OPEN_READONLY` rather than the constants as a group, which is a stylistic nit, not a correctness issue.
- [info] cli/src/nfs.rs:27-42 + 584-597 - Device-encoding helpers match glibc `gnu_dev_makedev`/`gnu_dev_major`/`gnu_dev_minor` exactly; the unit test covers `(0,0)`, small values, `(259, 65536)`, and `(0x12_345, 0x23_456)`, exercising the boundary between low-byte minor and high-bits minor, as well as 20-bit major upper bits. Adequate coverage for the bit layout.
