## Findings

- [low] sdk/rust/src/filesystem/hostfs_windows.rs:599-636 - The Windows-specific rename overwrite path is `fs::rename → on error: fs::remove + fs::rename`. There is a small race window where the source can disappear between the metadata check at line 574 and the destination removal at line 610-612: in that case the destination gets removed first and then the retry at line 617 fails with NotFound, so the caller sees "source missing" without learning that the destination was destroyed. The five rename tests cover the supported single-threaded scenarios, but cross-process modifications to a HostFS-backed directory can hit this. Document the trade-off (or, better, only remove the destination after a `fs::symlink_metadata(&old_path)` re-check) in a future polish PR — not blocking because the AgentFS Mutex single-threads server-side renames.
- [low] sdk/rust/src/filesystem/hostfs_windows.rs:351-357 - `chmod` calls `fs::metadata` + `fs::set_permissions`, both of which follow symlinks. The macOS HostFS at `hostfs_darwin.rs:573-590` deliberately uses `lchown` to avoid following. There is no direct `lchmod`/`lset_permissions` on Windows, but the Windows behavior of "chmod on a symlink chmods the target" differs silently from Linux/macOS. Consider either documenting this in the SDK FileSystem trait or rejecting chmod on symlinks. Low priority because the AgentFS NFS adapter rarely chmods symlinks.
- [low] sdk/rust/src/filesystem/hostfs_windows.rs:307-349 - `readdir` and `readdir_plus` call `fs::read_dir` synchronously inside async functions, whereas the Linux/macOS HostFS wraps them in `tokio::task::spawn_blocking`. For the NFS server's serialized request flow the practical impact is minor, but on busy directories the executor thread can stall briefly. Not a regression introduced by Phase 4, but a divergence worth recording for the next polish pass.
- [low] sdk/rust/src/filesystem/hostfs_windows.rs:682-687 - `validate_component` rejects empty / `.` / `..` / path separators but does not reject Windows reserved names (`CON`, `PRN`, `AUX`, `NUL`, `COM1-9`, `LPT1-9`) or reserved characters (`<>:"|?*`). Those will fail later as `io::Error` rather than `FsError::InvalidPath`. Acceptable for v1; would be a small enhancement.
- [low] sdk/rust/src/filesystem/hostfs_windows.rs:476-510 - `symlink` picks `symlink_dir` vs `symlink_file` based on `target_probe.is_dir()` at creation time. For dangling symlinks the default is `symlink_file`, which is irreversible on Windows. The trait doesn't expose a "type hint" to the caller, so this is the best v1 can do; worth a one-line note in the spike for the OverlayFS author to be aware of.
- [low] work/phase4-cli-smoke-windows-overlay-mount-nfs.log:14-23 - The overlay smoke confirms `Using overlay filesystem with base: \\?\F:\Tools\agentfs\work\phase4-overlay-base-smoke` is reached *before* the NFS Network Error 53, which satisfies the Phase 4 acceptance criterion. The remaining Error 53 is the same pre-existing Phase 0-3 Windows Client for NFS issue and is out of Phase 4 scope. No action required, called out for clarity.
- [info] work/phase4-windows-hostfs.md:82-89 + work/phase4-cargo-clippy-windows-sdk.log - `--all-targets` clippy fails on stale `benches/overlayfs.rs` and `benches/workload.rs` (path-based `OverlayFS` APIs that no longer exist post-inode-migration). - [info] sdk/rust/Cargo.toml:25-26 - `windows-sys = "0.61.2"` is added as a `cfg(target_os = "windows")`-only dependency with the minimum features (`Win32_Foundation`, `Win32_Storage_FileSystem`). Tight feature surface and Windows-only — Linux/macOS Cargo.lock updates are mechanical and the Linux SDK check passes. Good restraint.
- [info] sdk/rust/src/filesystem/hostfs_windows.rs:172-204 + 845-869 - The inode mapping uses a 63-bit FNV-1a fingerprint over `(VolumeSerialNumber, FILE_ID_128)`, bumps to 2 if it lands on 0 or 1, and resolves collisions via deterministic linear probing through `next_probe_inode`. ROOT_INO=1 is preserved. The `src_to_ino` + `inodes` bidirectional cache means lookup, getattr, readdir_plus, open, and NFS fileid3 conversion all converge on the same inode for the same Windows identity for the life of the HostFS instance. Matches the plan's "deterministic 63-bit fingerprint, reserve 1 for root, bidirectional cache with deterministic probe" requirements verbatim.
- [info] sdk/rust/src/filesystem/hostfs_windows.rs:725-767 - `query_file_info` opens an attribute-only handle with `FILE_FLAG_BACKUP_SEMANTICS | FILE_FLAG_OPEN_REPARSE_POINT` and asks for the share-everything mode, so it works for directories and for symlinks without following them. Preferred path: `GetFileInformationByHandleEx(FileIdInfo)` for the 128-bit FILE_ID_INFO; fallback: `GetFileInformationByHandle` for the 64-bit file index packed into the low 8 bytes of `[u8; 16]`. The 64-bit fallback collides with a preferred 128-bit identity only if the preferred high 8 bytes are zero on the same volume — vanishingly unlikely. Matches plan exactly.
- [info] sdk/rust/src/filesystem/hostfs_windows.rs:769-806 - `metadata_to_stats` synthesizes POSIX mode from Windows attributes: directory → `S_IFDIR | 0o777`, regular → `S_IFREG | 0o666`, symlink → `S_IFLNK | 0o666`, with read-only attribute masking off `0o222`. uid/gid/rdev are zero. Times are FILETIME-converted via `filetime_to_unix` (100ns ticks since 1601 → seconds + nsec since 1970). Sensible v1 mapping.
- [info] sdk/rust/src/filesystem/hostfs_windows.rs:359-363 + 456-474 - Conservative unsupported errors match the plan: `chown` always returns `Error::Internal("chown is not supported by Windows HostFS")`; `mknod` for non-regular file types returns `Error::Internal("mknod for special files is not supported by Windows HostFS")`; `mknod` for regular files falls through to `create_file`. No fake successes.
- [info] sdk/rust/src/filesystem/hostfs_windows.rs:818-836 - `time_change_to_filetime` rejects pre-1601 timestamps (returns `Error::Internal` for negative intervals), uses i128 arithmetic so a u64 FILETIME conversion can't overflow for any reasonable Unix timestamp, and threads `TimeChange::Omit` through as `None` → `null` pointer to `SetFileTime`, which is the documented "leave timestamp unchanged" signal. Correct.
- [info] sdk/rust/src/filesystem/hostfs_windows.rs:556-637 + tests 994-1066 - The Windows rename implementation handles four documented cases: same-path no-op, file-over-file overwrite via remove+retry, file/directory type-mismatch rejection (with destination preservation), and missing-source preservation. The three test functions (`test_hostfs_windows_rename_overwrite_file`, `test_hostfs_windows_rename_type_mismatch_preserves_destination`, `test_hostfs_windows_rename_missing_source_preserves_destination`) lock in the destination-preservation contracts the spike calls out.
- [info] sdk/rust/src/filesystem/hostfs_windows.rs:217-258 + 600-619 - Path-cache maintenance: `remove_cached_paths_under` (used on rename overwrite and after destructive remove+retry) clears both `inodes` and `src_to_ino` for any path under the renamed-out destination; `update_cached_paths_after_rename` rewrites all cached paths that match or are nested under `old_path` to point at `new_path`. Avoids the stale-path-cache-after-rename class of bug.
