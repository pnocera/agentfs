# Claude Review - Windows port implementation plan review

## Verdict

GO_STATUS: CHANGES_REQUESTED

## Findings

- [high] work/plan/implementation-plan.md:131-141 - Phase 4 Windows `HostFS` does not specify how Windows file identity (volume-scoped 128-bit `FILE_ID_INFO` or the legacy 64-bit `BY_HANDLE_FILE_INFORMATION.nFileIndex{High,Low}`) maps into the cross-platform `Stats.ino: i64` (see `sdk/rust/src/filesystem/mod.rs:108`). Without a deterministic, collision-safe mapping (e.g., a 64→i64 truncation/hash plus cache), `lookup`, `readdir_plus`, and the NFS adapter (`cli/src/nfs.rs:32`) will produce unstable inode IDs across reboots or across volumes that share low-order bits. This is the single hardest correctness item in the plan and needs to be designed before Phase 4 starts. Spell out the mapping strategy and how it interacts with the existing path/inode cache.
- [high] work/plan/implementation-plan.md:82-86 - Phase 2 proposes that `MountHandle::drop` "move out of the mounted drive before unmounting, using a safe directory such as `std::env::temp_dir()` rather than hardcoded `/`." The current implementation at `cli/src/mount/mod.rs:117` does `std::env::set_current_dir("/")` unconditionally for both Linux and macOS, and changing it for everyone is out of scope for the Windows port. Either cfg-gate the new behavior to Windows, or explicitly call the cross-platform change out as intentional (and Linux/macOS impact: `temp_dir()` is `/tmp` on most boxes, so it would still satisfy EBUSY avoidance, but the change should be documented and re-tested on Unix).
- [medium] work/plan/implementation-plan.md:184-191 - Phase 6 covers "agentfs mount list" but does not address `agentfs_sdk::get_mounts` at `sdk/rust/src/lib.rs:75-80`, which currently returns `vec![]` for every non-Linux target. `cli/src/cmd/mount.rs:list_mounts` (Linux only) and `cli/src/cmd/mount.rs:prune_mounts` (Linux only) consume `get_mounts()`. Either plan to implement a Windows version of `get_mounts` (parsing `mount.exe` output or, better, a CLI-owned registry of agentfs-tagged drives) or explicitly state that list/prune stay stubbed on Windows. As written, "use `umount.exe` for known agentfs drive mounts only" has no source of truth on Windows.
- [medium] work/plan/implementation-plan.md:36-53 - Phase 0's acceptance ("a manual NFS client command has been confirmed") is too soft for the load-bearing fact in the whole port: the exact UNC/port syntax for the Windows Client for NFS at a non-default port. Tighten the acceptance to "the captured working command line, including `\\127.0.0.1@PORT\!` vs `\\127.0.0.1:PORT\!`, is written into a notes file under `work/` and referenced by Phase 2." The current report only guesses (`work/windows-port-report.md:198,309`), and Phase 2 hardcodes against that guess.
- [medium] work/plan/implementation-plan.md:149-176 - Phase 5 does not address shell prompt / environment customization. The macOS template at `cli/src/cmd/run_darwin.rs:386-389` sets `PS1`/`ZDOTDIR` so the user sees the 🤖 prompt indicating they are inside an agentfs session. `cmd.exe`/PowerShell need a different mechanism (e.g., spawn with `/k prompt $G🤖$S` or pass `-NoExit -Command "function prompt {...}"`). Add a task for this or explicitly mark it deferred — without it the Windows session is silently indistinguishable from the host shell.
- [medium] work/plan/implementation-plan.md:165-168 - Phase 5 says "Set `AGENTFS=1`" and "use `none` or omit" for `AGENTFS_SANDBOX`. The existing convention (`cli/src/sandbox/darwin.rs:191` → `"macos-sandbox"`, `cli/src/sandbox/linux.rs:988` → `"linux-namespace"`) suggests an explicit non-empty value is preferred so downstream tools can detect the platform. Pick a deliberate name (e.g., `"windows-none"` or `"windows-overlay-only"`) and document the contract. "Set to none" is ambiguous between the literal string `"none"` and "do not set."
- [low] work/plan/implementation-plan.md:27 - "`cli/src/run_windows.rs` exists but only bails." The actual path is `cli/src/cmd/run_windows.rs` (verified via Glob). Same with `cli/src/cmd/run_not_supported.rs` at the sibling path. Fix the breadcrumb so a future reader can grep.
- [low] work/plan/implementation-plan.md:67-73 - Phase 1 says to replace `libc::{O_RDONLY, O_RDWR}` with "SDK-level open flag constants or local constants matching the existing POSIX values (`0` and `2`)." The `FileSystem::open` contract at `sdk/rust/src/filesystem/mod.rs:235` already documents flags as `libc::O_RDONLY`/`libc::O_RDWR`. Either define the constants in the SDK and migrate all callers, or pin the local constants in `cli/src/nfs.rs` with a comment explaining why we are not using `libc::`. As written the plan invites a half-migration where each platform interprets the same `i32` differently.
- [low] work/plan/implementation-plan.md:204-238 - Phase 7 only runs `cargo check` in Phase 0 and `cargo clippy` / `cargo test` in the final acceptance. Adding `cargo clippy -p agentfs --target x86_64-pc-windows-msvc --no-default-features` to Phase 1's acceptance would catch warnings earlier, before they accumulate across phases. The current acceptance only requires "reaches the command stubs or mount implementation work."
- [low] work/plan/implementation-plan.md:101-122 - Phase 3's "Reuse `cli/src/cmd/mount.rs` for Windows" is the right call, but `mount_nfs_backend` at `cli/src/cmd/mount.rs:199` currently does `args.mountpoint.exists()` and `std::fs::canonicalize(args.mountpoint.clone())?` before backend dispatch. On Windows, an unassigned `Z:` will fail both checks. The refactor must happen inside `mount_nfs_backend` (shared code), not just in the per-OS `mount(args)` entry point — call this out explicitly so it isn't missed.
- [low] work/plan/implementation-plan.md:185-187 - Phase 6 says "Ungate `opts.rs`, `main.rs`, and `cmd/mod.rs` from Unix-only to `any(unix, windows)`." For `cli/src/cmd/mod.rs:19-20` (`pub mod nfs`) and `cli/src/lib.rs:17-18` (`pub mod nfs`), this depends on Phase 1's libc cleanup having landed. State the dependency.

## Required Fixes

- Decide and document a Windows `Stats.ino` mapping strategy in Phase 4 (which Win32 API, how to fit 128 bits into `i64`, how to keep it stable across `lookup`/`readdir_plus` and the NFS adapter).
- In Phase 2, cfg-gate the proposed `MountHandle::drop` chdir change to Windows or explicitly mark the Linux/macOS behavior change as intentional with a re-test note.
- In Phase 6, state explicitly whether `agentfs_sdk::get_mounts` gets a Windows implementation or stays empty; tie `prune mounts` source-of-truth to that decision (do not unmount drives we cannot prove are ours).
- In Phase 0, tighten the acceptance: capture the working `mount.exe` command line (including non-default port syntax) into a `work/` notes file that Phase 2 must reference.
- In Phase 5, add a task for `cmd.exe`/PowerShell prompt customization (or explicitly mark deferred), and pick a concrete `AGENTFS_SANDBOX` value (matching the convention in `cli/src/sandbox/{darwin,linux}.rs`).
- Fix the path `cli/src/run_windows.rs` → `cli/src/cmd/run_windows.rs` in Current Checkout Facts.
- In Phase 1, either centralize `O_RDONLY`/`O_RDWR` constants in `agentfs-sdk` and migrate all consumers, or keep `libc::` on Unix and add a small Windows-only fallback module — pick one and write it down.
- In Phase 3, call out that the mountpoint-existence and canonicalize calls inside the shared `mount_nfs_backend` (not just per-OS entry points) must be refactored to allow unassigned drive letters on Windows.
