# Claude Review - Phase 0 baseline and Windows NFS client spike review

## Verdict

GO_STATUS: CHANGES_REQUESTED

## Findings

- [medium] work/windows-nfs-client-spike.md:138 - The recommended Phase 2 invocation hardcodes `timeout=1`. Windows `mount.exe` interprets `timeout` in *tenths of a second*, so `timeout=1` is 100 ms (default is 8 = 800 ms). Combined with `retry=1`, this will produce spurious mount failures against a localhost AgentFS NFS server under any TCP setup jitter, GC pause, or first-RPC stall, and the failure will look like a server/port-syntax bug rather than an aggressive timeout. The spike is the authoritative reference for Phase 2 (`work/plan/implementation-plan.md` line 95 says Phase 2 must implement the command from this artifact), so this needs to be corrected here before code copies it.
- [medium] work/windows-nfs-client-spike.md:182-190 - Phase 0 acceptance in the implementation plan (lines 55-56) requires that "the exact working command line is saved to `work/windows-nfs-client-spike.md`, including whether the non-default port form is `\\127.0.0.1@PORT\!`, `\\127.0.0.1:PORT\!`, or something else." No working command line was captured because no live AgentFS NFS server was running during the spike. The deferred runtime-probe mitigation in lines 125-131 is a defensible substitute, but the Phase 0 Conclusion (lines 182-190) calls Phase 0 prerequisites "satisfied" without stating that the live-server confirmation requirement is being explicitly deferred. The deferral should be promoted into the Conclusion so future reviewers and the Phase 2 implementer cannot miss it.
- [low] work/windows-nfs-client-spike.md:9 - "The Windows build baseline is clean" is accurate for what currently compiles, but the cleanness reflects that `nfsserve`, `nfs`, and `mount` are all gated out by `#[cfg(unix)]` at `cli/src/lib.rs:14-21`. The spike does not flag that the baseline therefore measures only the non-NFS surface; the *meaningful* Windows baseline will come after Phase 1 widens those gates. Worth a one-sentence caveat so the "clean" framing is not later cited as evidence that the NFS path already compiles on Windows.
- [low] work/windows-nfs-client-spike.md:107-122 - The interpretation that forms returning Error 53 are "syntactically accepted far enough" while `127.0.0.1:11111:/` returning Error 67 is "rejected as an invalid network name" is plausible but not conclusive: 53 = `ERROR_BAD_NETPATH`, 67 = `ERROR_BAD_NET_NAME`, and both are produced by the redirector / multiple-UNC-provider layer well before the NFS client parses any port suffix. The spike should either soften the interpretation ("error codes alone do not distinguish syntactic acceptance from name lookup failure") or add a control probe against a known-good UNC (e.g. `\\localhost\C$`) to compare the failure shape.
- [low] work/windows-nfs-client-spike.md:148-152 - Detecting the real Windows NFS `mount.exe` by matching the description string "Client for NFS export/share mount utility" is fragile across Windows language packs. Stronger signals: (a) `Get-AuthenticodeSignature` subject is Microsoft Windows, or (b) the binary is the one resolved from `%SystemRoot%\System32` (already the preferred path) *and* the running `NfsClnt` service is present. Document the language-pack fragility or list a fallback detector.
- [low] work/windows-nfs-client-spike.md:137-138 - `-o casesensitive=yes` is a defensible default for AgentFS semantics but flips Windows users into less-familiar behavior; the spike should note the rationale and how a user would override it, so Phase 3 user-facing docs inherit it.
- [low] work/phase0-cargo-check-windows-no-default-features.log:1 / work/phase0-cargo-check-windows-default-features.log:1 - Logs invoke `cargo check ...` (no `-p agentfs`) while the plan's Phase 0 tasks (lines 39-41) call for `cargo check -p agentfs ...`. Running from `cli/` is equivalent here (the package name is `agentfs` per `cli/Cargo.toml:2`), and the log lines `Checking agentfs v0.6.4 (F:\Tools\agentfs\cli)` confirm the right crate is exercised, but the spike should add a single sentence reconciling the invocation with the plan's `-p agentfs` form so a Phase 7 CI step does not regress to a different crate.
- [info] work/phase0-cargo-check-windows-default-features.log:1-9 - Second run uses warm cache and only re-checks the `agentfs` cli crate in 1.28 s; no transitive crate output is captured. Acceptable as a green signal, but if the goal of saving the log is to evidence the full Windows compile surface, the log was taken after a cache priming run and so does not exercise dependency compilation on a clean tree.

## Required Fixes

- Replace `timeout=1` in the Phase 2 recommended invocation (work/windows-nfs-client-spike.md:138) with a value calibrated for localhost NFS (`timeout=8`, i.e. 800 ms, matching the documented default) or omit it; pair with `retry=` >= 1 deliberately and explain the choice. If the spike intentionally wants aggressive failure for testability, say so and add a separate "production default" line.
- Update the Phase 0 Conclusion (work/windows-nfs-client-spike.md:182-190) to state explicitly that the plan's "exact working command line" Phase 0 acceptance is *not* met because no live AgentFS NFS server was available, and that confirmation is deferred to the Phase 2 runtime probe — link to the strategy in lines 125-131.
- Add one sentence near work/windows-nfs-client-spike.md:9 noting that the green Windows `cargo check` reflects `#[cfg(unix)]` gates excluding `nfsserve`/`nfs`/`mount` (see `cli/src/lib.rs:14-21`), and that the post-Phase-1 baseline is the one to evaluate the NFS path against.
- Soften the port-form interpretation (work/windows-nfs-client-spike.md:117-122) or add a control probe (e.g. `\\localhost\C$`) to distinguish redirector-level name-lookup failures from NFS-client port-syntax rejection.
- Note the language-pack fragility of the description-string detection heuristic (work/windows-nfs-client-spike.md:151) and list at least one fallback signal (signature / service presence).
